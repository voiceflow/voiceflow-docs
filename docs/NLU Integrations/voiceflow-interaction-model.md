## Data Structures

### Slot (Entity)

```typescript
interface Slot {
  key: string; // unique ID of the slot
  name: string; // name of the slot
  type: {
		value: string // built-in type of slot or "custom"
	};
	inputs: string[]; // examples/values of slot
  color?: string; // what color to render the slot as
}
```

The slot, also referred to as ŌĆ£entityŌĆØ on the frontend and other vendors, is a specific piece of an utterance (sentence) that we want to extract. For example, if I say *ŌĆ£I want a**small** **pepperoni** pizzaŌĆØ*, behind the scenes we define: `"I want a {size} {type} pizza"` so we know the `{size}` and `{type}` of pizza that a person wants as part of their order. Slots can either be custom, or prebuilt.

* `Slot.key`
  * this is the primary, unique ID of a slot within a model. All references to this slot in the data should ideally be done through the key, not the name. This is a 7-character random string generated by `cuid.slug`
* `Slot.name`
  * this is the name the user assigns to a slot. For example, `size` or `type` and this is how the slot is referenced on the UI for the user.
* `Slot.type.value`
  * examples: `Custom`, `VF.NUMBER`, `VF.EMAIL`, `VF.DATETIME`
  * We can define custom slots, like the pizza types: *ŌĆ£pepperoni, cheese, hawaiian, veggie, etc...ŌĆØ*, but there are also commonly used types of slots such as number, name, email. These are ŌĆ£predefinedŌĆØ or ŌĆ£prebuiltŌĆØ slot types, so the user doesnŌĆÖt have to give examples, it is already baked in and trained. Different NLP providers have different prebuilt slot types.
  * If it is not defined, we can just assume it is custom.
* `Slot.inputs`
  * for custom slots we need to give some examples. The format is an array of strings that are comma-separated values. The first entry in the CSV is the main example, and all trailing entries are synonyms. For example for my ***ŌĆ£mediumŌĆØ*** size - *ŌĆ£normalŌĆØ* and *ŌĆ£regularŌĆØ* are valid synonyms. This way when the user says *ŌĆ£regularŌĆØ* we can just standardize it as ***ŌĆ£mediumŌĆØ.***

```typescript
inputs: [
		"large, super, big, huge, massive", // large is the main example
		"medium, normal, regular",
		"small, petit, tiny, smaller"
]
```

* For prebuilt slots, we donŌĆÖt need to give inputs, so it can just be an empty array. But we can use this to give the model even more data points. For example, the VF.NAME prebuilt slot type might not be trained as well with ethnic names of the demographic the project is for, so we can add those as examples.

LetŌĆÖs put this all together and look at some examples of a slot:

### Intent

```typescript
// CUSTOM SLOT
{
	id: "vrg3y3e",
	name: "pizza size",
	type: { value: "Custom" },
	inputs: [
		"large, super, big, huge, massive",
		"medium, normal, regular",

	]
}

// PREBUILT SLOT
{
	id: "ujee3ybr",
	name: "customer name",
	type: { value: "VF.NAME" },

```

```typescript
interface Intent {
  key: string; // unique ID of the intent
  name: string; // name of intent
  inputs: { // list of utterances
		text: string; // 
	  slots?: string[]; // list of slots used within an utterance, used for frontend calculations, not important
		// ^ this can just be derived from the `text` itself
	}[];
	slots?: IntentSlot[]; // reference IntentSlot definition below
  noteID?: string; // attach notes/comments to intents, doesn't matter for ML
}
```

An intent is a generalization of a userŌĆÖs intention. There are so many ways to just say yes: ŌĆ£yesŌĆØ, ŌĆ£yeahŌĆØ, ŌĆ£sureŌĆØ, ŌĆ£okayŌĆØ, ŌĆ£no problemŌĆØ, and ŌĆ£very wellŌĆØŌĆ”.

There are a million ways a person could order pizza: *ŌĆ£I want some pepperoni pizzaŌĆØ, ŌĆ£may I have truffle pizza please sirŌĆØ, ŌĆ£yo man gimme some of that hawaiian pizzaŌĆØ.*\
But at the end of the day, they just want to order pizza. Intents can also contain slots (entities) to extract specific bits of information, like the type of pizza (*pepperoni, truffle, hawaiian*).

Intents can be either custom or prebuilt. Some intents (Yes, No, Stop, etc.) are commonly used, that they are offered prebuilt without the need for user to defined examples.

* `Intent.key`
  * this is the primary, unique ID of a intent within a model. All references to this intent in the data should ideally be done through the key, not the name.
    * For custom intents, this is a 7-character random string generated by `cuid.slug`
    * For prebuilt intents, the key is actually the type, and the same as the name. Each type of prebuilt intent can only exist once and must be unique.
      * e.g. in a Voiceflow-NLP project, it would be `VF.YES`, in an Alexa-type project, the `key` and `name` of that prebuilt would be `AMAZON.YesIntent`,
* `Intent.name`
  * this is the name the user assigns to an intent. A custom one could be ŌĆ£order pizzaŌĆØ or ŌĆ£transfer moneyŌĆØ describing the usage. Prebuilt intents have unique, predefined names that are the same as the `key` such as `VF.YES` or `VF.STOP`
* `Intent.inputs`
  * Examples of what the user could say to trigger this intent, as well as slots defined within each example. Each input contains a `text` and `slots` field.\
    To capture the user saying *ŌĆ£I want some pepperoni pizzaŌĆØ.* The input would look like this:

```typescript
{
	text: "I want some {{[pizza type].46173ucs}} pizza",
	slots: ["46173ucs"]
}
```

Note that slots inside the `text` field take on a `{{[<SLOT NAME>].<SLOT ID>}}` format. They can be retrieved with this regex: `/{{\[(\w{1,32})]\.(\w{1,32})}}/gi`

`inputs[].slots` is just a utility field to quickly know what slots each utterance uses, it is not recommended to use this field for any practical purposes. 

* Usually, for prebuilt intents, `inputs` is just an empty array, but can be extended to provide additional context. If there is some unique way of saying *ŌĆ£yesŌĆØ* like *ŌĆ£cool manŌĆØ* that the projectŌĆÖs audience uses, and the NLP is not picking it up, we can explicitly define it as an example.
* `Intent.slots`
  * metadata for how specific slots are used within an intent, seed the **IntentSlot** section.

Let's put all this together and see some examples of an intent:

```typescript
// REGULAR INTENT
{
	key: "vo183uth",
	name: "order pizza",
	inputs: [{
		text: "I want some {{[pizza type].46173ucs}} pizza",
		slots: ["46173ucs"]
	}, {
		text: "I want some pizza",
		slots: []
	}],
	slots: [{
		id: "46173ucs",
		required: false,
		dialog: { ... }
	}]
}

// PREBUILT INTENT
{
	key: "VF.YES",
	name: "VF.YES",
	inputs: [],
	slots: [],
}
```

### IntentSlot

This is used for entity fulfillment, which is not a supported feature on all NLPs

```typescript
interface IntentSlot {
  id: string; // the Slot.key that this entry references
	required: boolean;
  dialog: {
	  utterances: { // this is exact format same as intent.inputs defined above
			text: string,
			slots?: string[]
		}[];
		prompt: Object[]; // bot responses to ask the user for the slot

		// not used outside of Alexa
	  confirm: Object[]; // bot responses for confirming the value of a slot
	  confirmEnabled: boolean;
	};
}
```

The **IntentSlot** is metadata about how a specific **Slot** is used within the context of an intent. In the ŌĆ£Order PizzaŌĆØ intent the user could say *ŌĆ£I want a pizzaŌĆØ, ŌĆ£I want a**pepperoni** pizzaŌĆØ* or even ŌĆ£*I want a**small** **pepperoni** pizzaŌĆØ.*  We might want to ensure that the `{type}` slot is filled out before we proceed with the conversation, but it's okay if they donŌĆÖt say the `{size}` since we can just assume itŌĆÖs ŌĆ£mediumŌĆØ size if unfilled. 

When we want to make sure a slot is filled even when the user didnŌĆÖt initially say it, we define that flow within `IntentSlot`.

* `IntentSlot.id`
  * the reference to the specific slot that this metadata is for. This is the `Slot.key`
* `IntentSlot.required`
  * whether this slot must be filled before the intent resolves, **if`required` is `false`, nothing in `IntentSlot.dialog` really matters.**
* `IntentSlot.dialog.prompt`
  * when the slot is not filled, what should we ask the user? The type of this depends on the **type** of project. This could be some text or a voice SSML prompt. Randomly select one entry to ask.
* `IntentSlot.dialog.utterances`
  * example utterances of what the user could say to fulfill a required slot. This is identical in format to `Intent.inputs`

Here is a example of an entity fulfillment conversation flow given the following intent:

```typescript
// INTENT
{
	key: "vo183uth",
	name: "order pizza",
	inputs: [{
		text: "I want some pizza",
		slots: []
	}],
	slots: [{ // INTENT SLOT
		id: "46173ucs", // we are looking for the "pizza type" slot
		required: true, // this must be filled out
		dialog: {
			utterances: [{ text: "{{[pizza type].46173ucs}} please", slots: ["46173ucs"] }],
			// text type project:
			prompt: [{content: [{children: [{ text: "what type of pizza?" }]}]}]
			confirmEnabled: false,
			confirm: [],
		}
	}]
}
```

ŌåÉ ŌĆ£what type of pizza?ŌĆØ (`intentSlot.dialog.prompt`)

ŌåÆ ŌĆ£pepperoni pleaseŌĆØ (`intentSlot.dialog.utterances`)

ŌåÉ ŌĆ£got it, ordering a pepperoni pizzaŌĆØ (maybe a follow up speak step)

# Model

A model is just a list of intents and slots.

```typescript
interface Model {
	intents: Intent[];
	slots: Slot[];
}
```

On any given project, there are 3 places you could potentially find an NLU Model. WeŌĆÖll go in-depth on each one of these.

### `version.platformData`

This is the direct model that the user edits on the frontend. This model is tied to NLP vendor of the project (i.e. Google Actions, Alexa, LUIS, Rasa) and will contain special prebuilt slots and intents from those platforms.

* For example, on an Alexa project, there might be [built-in intents](https://developer.amazon.com/en-US/docs/alexa/custom-skills/standard-built-in-intents.html) with `Intent.key` & `Intent.name` that is `AMAZON.YesIntent`, or built-in [slot types](https://developer.amazon.com/en-US/docs/alexa/custom-skills/slot-type-reference.html) such as `AMAZON.DATE` or `AMAZON.PhoneNumber` is their `Slot.type.value`.
* On a Voiceflow project, the `Intent.key` and `Intent.name` might be `VF.YES`, and built in slots types such as `VF.DATETIME` or `VF.NAME` (`Slot.type.value`)

All entries listed in `version.platformData` can be directly seen in the interaction model manager and modified by the user.

This model can have some exceptions when pushed to external platforms, read the **ŌĆ£On PublishŌĆØ** section.

### `version.prototype.model`

This is the model generated after compile-time and is always guaranteed to be in the Voiceflow NLP format, with Voiceflow prebuilt intent and slots. Every time the user starts the prototype tab, we are regenerating this model and updating it, derived from the model on `version.platformData`.

For Voiceflow Chat/Voice projects, this is generated entirely on `general-service` with the `/prototype/renderSync` endpoint, the **translating** step below is not applied.

For Alexa, Google Actions, and Dialogflow projects, their models go through the **translating** and **pruning** steps on their respective services, `alexa-service`, `google-service` with a `/renderSync` endpoint as well, and then send the updated model to `general-service`'s `/compile` endpoint.

We apply several different transformations to the original `version.platformData`:

* **translating**: If the `version.platformData` uses any non-Voiceflow NLP, converts all their built-in intents and slots to the Voiceflow equivalent:
  * e.g. The intent `AMAZON.YesIntent` becomes `VF.YES`
  * e.g. The slot type `[AMAZON.DATE](http://AMAZON.DATE)` becomes `VF.DATETIME`
    If there is no equivalent, it just goes into `version.prototype.model` as is.\
    
* **pruning:** all non-used intents (if they are not referenced in an intent, button, or choice step), are deleted, and all non-used slots are also removed. So if I have an ŌĆ£order pizzaŌĆØ intent but it is not actually referenced by any step in the project, it does not appear in `version.prototype.model`.\
  This helps the final prototype be much more accurate by not training on intents that will never trigger anything.
* **adding prebuilt examples**:

    This is for the sake of LUIS which doesnŌĆÖt have prebuilt intents. These are the generic examples so that users donŌĆÖt have to define them and can quickly use the intent as a ŌĆ£prebuiltŌĆØ.
  * For intents, we have a [gigantic list of example phrases](https://github.com/voiceflow/libs/blob/11d66c21b5c739a541152199f40825d5baeb3b57/packages/voiceflow-types/src/constants/intent.ts#L26). We can merge this any extension examples the users have given
